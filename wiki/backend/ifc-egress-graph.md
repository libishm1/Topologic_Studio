# IFC Egress Graph

This chunk covers the graph generated by `POST /ifc-egress-graph` in `topologicpy-web-backend/app/main.py`.

## Inputs

The frontend sends arrays of browser-extracted geometry:

- `floors`: IFC slab-like geometry.
- `stairs`: IFC stair and stair-flight geometry.
- `doors`: IFC door geometry for forced waypoints.
- `walls`: IFC wall geometry for path obstacles.

Important request parameters:

- `use_walls`: keep wall segments for path blocking.
- `agent_height`: offset applied to sampled walkable points.
- `base_spacing`: floor sampling control, clamped to at least `1.5`.
- `max_edge_floor`, `max_edge_stair`: connection thresholds.
- `up_axis`: frontend currently sends `"y"`.
- `rectilinear`: distance graph with diagonal suppression.
- `grid_snap`: voxel/cardinal graph option.
- `grid_cell_size`: grid-snap cell size.

## Pipeline

1. Validate that at least floors or stairs are present.
2. Estimate maximum model height and exclude triangles close to the top to reduce roof-as-floor mistakes.
3. Sample stair points first because stair connectivity is usually the fragile part.
4. Use remaining budget for floor points.
5. Extract door bottom-center points before applying the agent-height offset.
6. Apply the agent-height offset to all sampled and door points.
7. Build adjacency by either hybrid distance matching or grid-snap rectilinear cells.
8. Extract wall segments, but do not delete display graph edges.
9. Force-connect door nodes to up to five nearby floor nodes within `3.0` units.
10. Store the result in `LAST_GRAPHS["ifc"]`.

## Design Details

- The graph display intentionally keeps all edges; walls are enforced during route search. This keeps visual debugging possible.
- Door nodes are exempt from wall blocking in path search because doors are openings through wall geometry.
- Grid-snap mode reindexes nodes, so the code resolves door nodes back to grid UIDs after snapping.
- Stair vertical cell size is capped at `0.15` in grid-snap mode to preserve tread-level connectivity.
- The response includes both geometry edges and `edge_ids` so the frontend can recolor edges by node temperature later.

## Current Response Shape

See [api/ifc-egress.md](../api/ifc-egress.md). The response includes:

- `mode: "ifc"`
- `stats.nodes`
- `stats.edges`
- `stats.door_nodes`
- `stats.wall_segments`
- `edges`
- `edge_ids`
- `coords`

## Implementation Risks

- The frontend sends `up_axis: "y"`, but `IfcEgressRequest` defaults to `"z"`. New clients must be explicit.
- Agent-height offset is applied to `p[2]` regardless of `up_axis`. With the current frontend transform this may be intentional visually, but it is a risk if `up_axis` becomes configurable.
- `_nearest_node_id` prefers same-floor matching using Y-distance. That matches the current frontend but should be generalized before supporting arbitrary axes.
- `LAST_GRAPHS["ifc"]` is global process memory, so concurrent users can overwrite each other's graph.
